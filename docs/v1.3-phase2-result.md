# Landfall v1.3, Phase 2 result — the fixed-ROI coverage gap in large track-offset counterfactuals

## What was being tested

Continuing the session's "try more edge cases" pattern after adding a 4th storm
(`docs/v1.3-phase1-result.md`): two angles not yet stress-tested were prompt-injection-
style input to the compiler, and extreme-but-schema-valid counterfactual physics
(large `track_offset_km`, up to the compiler's own stated max of 500 km).

## Compiler: clean result

Adversarial and degenerate inputs — an explicit "ignore previous instructions" prefix,
an attempt to extract the system prompt, an injected fake "SYSTEM: new rule" trying to
whitelist an unregistered storm name, a request to dictate raw out-of-range JSON
directly, an empty string, whitespace-only input, and gibberish — were all handled
correctly: every injection attempt was either refused outright or had only its valid
component compiled, with no leaked instructions and no crash on any degenerate input.
This matches `compiler.py`'s own stated design: the system prompt is not the security
boundary, `ScenarioConfig`'s pydantic validation is, and that boundary held.

## Physics: a real "fail loud, not plausible" violation, found and fixed

Pushing a track's offset toward the compiler's stated max (500 km) produced a
suspicious result: `mangkhut +500km@180°` returned **$0.00 damage but 137,355 affected
population** — not an obvious crash, but an internally inconsistent-looking pair of
numbers worth checking before trusting either.

**Root cause, traced to the actual computation, not assumed.** `landfall/hazard/wind.py`'s
`wind_field()` builds its centroid grid strictly from `roi_centroids(bounds)`, where
`bounds` is the storm's *static*, registry-fixed ROI box (`landfall/storms.py`) — sized
once, to the *historical* track. It is never re-derived, expanded, or re-centered for a
counterfactual. A large enough `track_offset_km` can therefore push the perturbed
track's core (or all of it) outside that fixed box, and the simulation would silently
compute over a grid that never contains where the storm actually went.

**Confirmed directly at the wind-field level**, not inferred from the damage number
alone: for Rolly at `track_offset_km=300, bearing=0`, `wind.intensity.max() == 0.0`
across the *entire* ROI grid — the perturbed track's latitude range (16.2°–19.4°) had
moved entirely past the ROI's `lat_max=15.0`. This is a hard coverage cliff, not a
smooth physical decay: at 150km the track already clips only the box's edge (69 m/s
present, but $0 damage — a corner of the box with negligible underlying exposure); by
300km there is no wind anywhere in the grid at all.

**A sweep across all four storms (bearing 0/north) found this varies drastically by
ROI box size:**

| Storm | ROI box (lon×lat) | Damage hits exactly $0 at offset | % of compiler's 500km max |
|---|---|---|---|
| Rolly | 3.5°×3° (smallest) | **150 km** | **30%** |
| Mangkhut | 4.5°×4° | 300 km | 60% |
| Haiyan | 5°×5° | 400 km | 80% |
| Odette | 8°×4° (largest) | 400 km | 80% |

Rolly is the most exposed: a user asking a perfectly reasonable counterfactual —
"shift Rolly's track 200 km north" — is well inside the schema's valid range and gets
back a confident-looking $0 damage / 0 affected population, groundedness-verifier-clean
(0 *is* the cached number), that actually means "the grid never covered where this
storm went," not "this shift causes no damage." This is precisely the failure mode PRD
§4.2's "fail loud, not plausible" principle exists to prevent — previously applied only
to the compiler's input schema, not to the physics layer's own silent assumptions.

## Fix

Added `ROICoverageError` (`landfall/impact/engine.py`) and a `_check_roi_coverage()`
check run immediately after `wind_field()`, before the (expensive) exposure/impact
calculation: if `wind.intensity.max() == 0` anywhere in the storm's ROI grid, `run()`
raises a clear, human-readable refusal instead of silently returning a zero-filled
result. `cli.py`'s `cmd_run` and `cmd_narrate` catch it and print `Refused: ...`,
matching the compiler's own refusal-presentation convention, instead of a raw
traceback.

The check is deliberately narrow — it fires only on a *complete* absence of wind
anywhere in the grid (`== 0`, not "below some damage threshold"), so it does not
trip on legitimate low-damage counterfactuals where wind is genuinely present but
mild. Verified this doesn't produce false positives: all four storms' historical
baselines (offset=0) and a modest in-bounds counterfactual (Rolly +50km@90°, $34.2M)
compute normally; only the confirmed all-zero cases (e.g. Rolly +300km@0°) are refused.

## What this doesn't resolve

- The fixed ROI is still fixed — this catches the *complete* failure case (track fully
  outside the grid) but does not fix the underlying gap for a track that's *partially*
  outside the box (e.g. Rolly at 100–150km, where a sliver of the box still sees wind
  but the storm's real population-dense path has mostly left it). Those cases still
  silently under-report, just not down to a hard, checkable zero. Re-deriving the ROI
  per-scenario (expanding the grid to cover the perturbed track, then re-fetching
  exposure for the new box) would close this fully, but is a materially bigger change
  — new centroids, a fresh LitPop/OSM exposure fetch per request — not attempted here.
- No change to the compiler's stated `[0, 500]` km range or its documentation of what
  "meaningful" means; the schema still accepts these requests, they're now refused at
  the physics layer instead of returning a misleading number.
- Re-verified E2 (no scenario variant in `evals/e2_groundedness.py`'s fixed set trips
  the new guard) and E3 (compiler eval unaffected, engine-side change) — see commit for
  numbers.
