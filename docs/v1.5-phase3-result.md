# Landfall Viz v1.5, Phase 3 result — per-timestep cumulative damage

## Why this phase exists

Phase 4 of `landfall-viz-prd.md` (§7) is a time-dynamic replay: a scrubber animating the
storm's progression, wind swath painting in, damage columns accumulating. That needs a
number the engine has never computed — damage *as a function of time*, not just the
final total. Phase 3 is that engine addition, gated on its own exit test before any
animation code exists (§6: "the final timestep's cumulative damage must equal the
existing single-pass E1 result within rounding, for all four storms").

## The one real risk, and how it was resolved

Damage is not additive over time: a building destroyed at landfall is not destroyed
again an hour later. Summing per-timestep damage would double-count and could never
reconcile with the validated single-pass total. The only definition that is both
physically correct and reconciles by construction:

> **Cumulative damage at frame _i_ = `ImpactCalc` applied to the elementwise running
> max of the wind field over track positions `0..i`.**

Per-cell wind only ratchets upward under a running max, so cumulative damage is
monotonic non-decreasing, and the final frame's wind field — the running max over
*all* resampled positions — is mathematically the same field `hazard.wind.wind_field()`
already produces for the single-pass total. Reconciliation is therefore an identity,
not a coincidence: if it failed, that would mean the two code paths compute different
fields, a real bug, not a rounding question.

This is deliberately **not** instantaneous-damage-summed-over-time — that approach was
considered and rejected in the design; the code never implements it.

## Frame cadence

`src/landfall/impact/timeseries.py`: subsample the ROI-transit window (track positions,
at the v1.4 0.5 h resampling, where the storm produces any ≥17.5 m/s wind in the ROI) at
stride `N = ceil(window_len / 80)`, capping the frame count at 80. The **final frame is
always forced to the running max over all resampled positions**, not just the
subsampled ones — so a storm whose true peak sits between two sampled frames still
reconciles exactly; subsampling can only affect which *intermediate* frames exist, never
the reconciling one.

| Storm | Frames | Transit window (ROI-local) | Notes |
|---|---|---|---|
| Haiyan | 48 | 2013-11-07 14:30 → 2013-11-08 14:00 (24h span) | |
| Mangkhut | 69 | 2018-09-14 06:00 → 2018-09-15 16:00 (34h span) | longest transit, most frames |
| Odette | 45 | 2021-12-15 21:30 → 2021-12-17 17:00 (44h span) | |
| Rolly | 35 | 2020-10-31 14:30 → 2020-11-01 07:30 (17h span) | shortest transit, fewest frames |

All four land inside the ~40–80 target band the design aimed for; no storm needed the
cap floor (window this small) or hit the 80-frame ceiling.

## Monotonicity

`test_timeseries_cumulative_damage_is_monotonic` checks every consecutive frame pair,
for the ROI total *and* for every individual municipality, in all four storms: **zero
violations**. Cumulative damage never decreases — consistent with a running-max-driven
definition and inconsistent with any computation path that could "undo" damage already
counted.

## Scope decisions (as designed, not deviations)

- **Point estimate only** (`q=0.5`) is animated per frame. The EDR low/high range
  (`q=0.25/0.75`) stays a single-pass, whole-storm uncertainty figure in `damage.json`'s
  existing `total_damage_usd_range` — recomputing it per frame would triple the CLIMADA
  runs per frame for a number Phase 4 shows statically, not animated.
- **Affected population** is likewise single-pass only, out of Phase 3 scope — it isn't
  a running-max quantity in the same sense damage is.
- Per-municipality cumulative damage *is* computed and exported per frame (needed for
  Phase 4's per-municipality column animation), reusing the existing
  `impact.municipality.damage_by_municipality` join.

## What this phase added

- `src/landfall/hazard/wind.py`: factored `resample_track_hazard()` (the v1.4 densify
  step, now shared) and added `wind_field_timeseries()` (`store_windfields=True`,
  returns per-position windfields instead of only the collapsed max). `wind_field()`'s
  existing public behavior is unchanged.
- `src/landfall/impact/timeseries.py` (new): `run_timeseries()` / `run_timeseries_baseline()`,
  cached under `data/cache/timeseries/` (gitignored), keyed on the existing impact
  `_cache_key` plus `max_frames` (a cadence change invalidates the cache the same way a
  calibration change invalidates the impact cache).
- `src/landfall/export_viz.py`: `timeseries.json` (frames array, copied verbatim from
  the timeseries cache — same no-new-numbers discipline as `damage.json`) and
  `wind/frames.json` (per-frame running-max wind grids, same format as `max_swath.json`)
  per bundle; `meta.json` gained a `timeseries` block recording availability, frame
  count, cadence rule, and both reconciliation diffs. `export-viz --no-timeseries` opts
  out of the (larger, slower) timeseries export for callers that only need Tier 1.
- `tests/test_export_viz.py`: 5 new tests — the two reconciliation checks (total,
  per-municipality), monotonicity, wind-grid final-frame identity, and meta-block
  presence. 27 tests pass total.

## Bundle size — a real cost, disclosed rather than hidden

Timeseries data is large relative to the Tier-1 bundles:

| File | Haiyan | Rolly | Odette | Mangkhut |
|---|---|---|---|---|
| `timeseries.json` | 3.7M | 0.8M | 3.5M | 2.8M |
| `wind/frames.json` | 4.0M | 1.2M | 4.9M | 4.3M |

Total `viz/public/data/` is **28M** with all four storms' timeseries included (was a few
MB for Tier 1 alone). This is the bundle-size risk `landfall-viz-prd.md` §8 already
names ("Bundle sizes bloat the repo... Git-LFS threshold decided in Phase 1"); Phase 1
judged native-resolution single grids small enough to commit directly, but a full
per-frame animation multiplies that by frame count. **Not resolved in this phase** —
flagged for a decision before Phase 4 ships: commit as-is (28M is not large for a git
repo), move `wind/frames.json` to Git-LFS, or reduce per-frame wind-grid resolution
(coarser than the native 150 arcsec used for the static swath) since the animation is
read at a glance rather than inspected cell-by-cell. `damage` frames are small text
(JSON) either way; `wind/frames.json` is the dominant cost.

## What this phase doesn't do

- No viz/frontend animation code — Phase 4 (scrubber, painting wind, accumulating
  columns) is a separate, future step that consumes `timeseries.json` and
  `wind/frames.json`.
- No change to `engine.py`'s `run()` — it remains the single validated source of truth
  that Phase 3 reconciles against, untouched.
- No IMPF calibration, coastal-filter, or fragility-curve changes.
