# Landfall v1.2, Phase 1 result — reporting genuine calibration uncertainty instead of a doomed point estimate

## The problem this responds to

v1.1 Phase 5 proved the underlying issue was structural, not a specific bug: swapping
the WP2 impact function's single `v_half` from TDR's 188.4 to RMSF's 84.7 multiplies
every storm's damage total by a near-uniform ~16–19×, regardless of the storm's
specific wind-speed distribution. Applied to three storms whose pre-existing TDR-era
deficits (18.6×, 575×, 2.5–5× under) were too far apart, that fix necessarily
overshoots the shallowest one (Odette). **No single national v_half can close all
three simultaneously** — this is arithmetic, not a search problem with a better answer
waiting to be found by trying more calibration approaches.

## Two things tried; one is a negative result

**Tried and ruled out: filtering to "clean wind-only" events.** Eberenz et al. 2021's
`EDR` calibration approach fits `v_half` individually to 83 historical WP2 (Philippines)
typhoons (1980–2016) and tags each with hazard-type flags (`Surge`, `Rain`, `Flood`,
`Slide`, `Other`) — including Haiyan itself (`ibtracsID 2013306N07162`, individually
fitted `v_half=50.9`, flagged `Surge=True`, consistent with Haiyan's well-documented
storm-surge-dominated death toll). Hypothesis: filtering to events with none of these
flags set (n=47 of 83) — i.e. damage plausibly wind-dominated — should give a cleaner,
more wind-appropriate `v_half` than fitting across all 83 events regardless of cause.
**Result: median `v_half=81.0` for the clean subset, nearly identical to RMSF's 84.7
already in use, and not statistically distinguishable from the 36 hazard-flagged events
(Mann-Whitney U test, p=0.16).** The per-event variance (std ≈ 95–100 m/s in both
subsets) swamps any signal from removing surge/flood-contaminated events. This isn't
where the problem lives — filtering the same dataset a different way doesn't produce a
materially different number.

**What actually works: report the per-event distribution's spread, not a single point
from it.** The same EDR dataset that individually fits each historical event's
`v_half` already contains the genuine calibration uncertainty for this region — 83
values ranging from 25.8 to 325.7 m/s. CLIMADA exposes any quantile of this
already-bundled, already-published distribution via
`ImpfSetTropCyclone.from_calibrated_regional_ImpfSet(calibration_approach="EDR", q=...)`.
Computing Haiyan/Rolly/Odette's damage at q=0.25 and q=0.75 (interquartile range) using
the *current* pipeline (v1.1 Phase 3's hybrid exposure for Rolly, unchanged hazard
config):

| Storm | q0.25 (high damage) | q0.5 (point estimate) | q0.75 (low damage) | NDRRMC actual | Bracketed? |
|---|---|---|---|---|---|
| Haiyan | $2,891.2M | $695.4M | $49.6M | $917M | **✓** |
| Rolly | $456.8M | $81.4M | $5.4M | $233M | **✓** |
| Odette — held out | $23,530.7M | $3,095.2M | $185.1M | $459–915M | **✓** |

**All three storms' actual recorded damage falls inside the interquartile range —
including Odette, the storm every point-estimate approach across v1.1 broke.** This is
the first thing tried across the whole v1.1/v1.2 effort that works for all three
simultaneously, held-out storm included.

This is not a new fragility curve — CLAUDE.md's hard non-goal ("no custom fragility
curves — published/calibrated impact functions only") is unaffected. It's a different,
still-published quantile selection from calibration data CLIMADA already ships and this
project already cites. The point estimate itself moved slightly too, as a side effect
of switching from RMSF (v_half=84.7) to EDR's median (v_half=87.6, nearly identical) for
internal consistency with the range mechanism — Haiyan and Rolly's point-estimate error
factors got very slightly worse (1.18×→1.32× under; 2.54×→2.86× under) while Odette's
got slightly better (3.9–7.7×→3.4–6.7× over). Minor in either direction; the range, not
the point estimate, is this phase's actual claim.

## The honest catch: the ranges are wide, and one bound is not a serious estimate

Odette's range spans $185.1M to $23,530.7M — roughly 127× top-to-bottom, and $23.5B is
not a plausible damage estimate for any Philippine typhoon on record; it's a
mathematical artifact of how far the per-event `v_half` distribution's low tail (down
to 25.8 m/s, `v_thresh` itself) extrapolates once applied to Odette's specific exposure.
Reporting it anyway, un-trimmed, is the choice made here: capping or trimming the range
to "look more credible" would mean deciding in advance which parts of a genuinely messy,
published uncertainty distribution to hide, which is a worse failure mode than an
uncomfortably wide number. The width itself is the finding — it says a wind-only model
calibrated this way genuinely cannot narrow Philippine typhoon damage down to better
than roughly two orders of magnitude for some storms, and pretending otherwise with a
tighter, invented-looking band would be less honest than this.

## What changed in the codebase

`landfall/impact/engine.py`: `IMPF_CALIBRATION_APPROACH` switched from `"RMSF"` to
`"EDR"` (median, `q=0.5`, for the point estimate — `IMPF_POINT_QUANTILE`); a new
`IMPF_RANGE_QUANTILES = (0.25, 0.75)` constant drives two additional `ImpactCalc` runs
per scenario, added to the cached result as `total_damage_usd_range: {"low", "high"}`.
`total_damage_usd` (the point estimate) is unchanged in shape, so the narrator,
verifier, CLI, and E2 eval all continue to work against it without modification —
verified by smoke-testing `landfall narrate haiyan` end to end (3/3 groundedness,
unchanged). The scenario cache was cleared and regenerated, same as every prior phase
that changed the impact-function configuration (the cache key doesn't cover physics
config, only the perturbation).

## What this phase deliberately doesn't touch

- **The narrator and verifier still speak in a single point-estimate number.** The
  range exists in the cached data and can be surfaced in a future narration/CLI update,
  but doing so changes the narrator's prompt, the groundedness verifier's reference-value
  handling (two numbers instead of one), and the CLI's output format — a real, separate
  piece of work, not silently bundled into this phase.
- **Per-municipality breakdown is still point-estimate only** (median quantile) — the
  range is a total-damage-only figure in this pass, not computed per municipality.
- **The E2 eval and E1 waterfall's historical Phase-1-through-4 columns are unchanged**
  — this phase only changes what the *current* (rightmost) column reports going
  forward; it doesn't retroactively rewrite v1.1's already-committed history.
