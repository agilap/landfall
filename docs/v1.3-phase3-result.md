# Landfall v1.3, Phase 3 result — the scenario cache trusted a stale calibration silently

## What was being tested

Continuing this session's edge-case pattern after the 4th storm (Phase 1) and the
ROI-coverage guard (Phase 2): the scenario disk cache itself — "every scenario run is
cached to disk keyed on a scenario-config hash" (`CLAUDE.md`'s own engineering rule) —
had never been stress-tested for whether that key actually captures everything that
determines the cached output.

## The bug, confirmed by tampering the cache directly

`ScenarioConfig.scenario_hash()` (`landfall/scenario.py`) hashes only the user-facing
scenario fields: storm key, track offset, bearing, intensity delta. `run()`
(`landfall/impact/engine.py`) used this hash, alone, as the cache filename.

But the calibration approach/quantiles (`IMPF_CALIBRATION_APPROACH`,
`IMPF_POINT_QUANTILE`, `IMPF_RANGE_QUANTILES`) and each storm's ROI bounds
(`landfall/storms.py`) also determine the output — and both are mutable, non-hashed
inputs. Directly confirmed the consequence: took a real cached result for Haiyan,
hand-edited its `total_damage_usd` to an obviously-fake `1.23` and its
`calibration_approach` to `"TDR"` (the pre-v1.1 methodology), and called `run()` again
for the *identical* `ScenarioConfig`. **It returned the tampered value unchanged** —
`run()` has no way to tell "this file was computed under the current code" from "this
file is stale."

This is not a hypothetical. This project's calibration approach has genuinely changed
twice (TDR → RMSF in v1.1 Phase 1, RMSF → EDR in v1.2 Phase 1), and this same session's
Phase 2 added an ROI-coverage guard — I personally had to manually
`rm data/cache/scenarios/*.json` to test it, because a stale pre-guard cache entry
would otherwise have silently kept returning the old (undetected) zero-damage result.
Nothing in the code enforces that step; it only happened because I happened to
remember it while testing something else.

## Fix

Added `_cache_key()` (`landfall/impact/engine.py`), which folds
`scenario.scenario_hash()` together with the calibration constants and the storm's
`roi_bounds` into one fingerprint, and used it as the cache filename instead of the
bare scenario hash. A future change to calibration or ROI bounds now naturally produces
a *different* key — old entries go quietly unreferenced rather than being silently
served under a methodology the current code no longer uses.

Re-ran the exact tampering test after the fix, plus a stronger variant: planted the
tampered file at the *old-style* key (`scenario.scenario_hash()` alone, simulating what
a genuinely stale pre-fix cache entry looks like) and called `run()` again — it
correctly ignored the old-style file entirely and recomputed fresh, returning the real
$695,357,863.53, not the tampered $1.23. (Tampering the file at the *new*,
correctly-fingerprinted key is still trusted — that's the accepted, by-design
remaining trust boundary: "what's on disk for the exact current computation fingerprint
is trusted," which is what a disk cache is supposed to do; the fix specifically closes
the "computation changed but the same file gets reused" gap, not "someone hand-edits
your local disk.")

Pinned with a new regression test file, `tests/test_engine_cache.py` (3 cases: cache
key changes with calibration/ROI fingerprint; a stale old-style cache entry is not
served; the correctly-fingerprinted cache still works as a cache). Unlike
`tests/test_groundedness.py`, this one requires one real engine computation for setup
(CLIMADA compute, ~15–20s) rather than being a pure/instant unit test, but still no API
calls.

## Verified, no regressions

- All four storms' historical baselines recompute to the exact same values already
  verified earlier this session (Haiyan $695,357,863.53, Rolly $81,398,985.18, Odette
  $3,095,153,114.86, Mangkhut $125,983,187.57).
- E3: 48/48 exact-config, 36/37 rejection — unaffected (engine-side change only), same
  known Mangkut miss.
- E2: 336/336 = 100% final groundedness — unaffected; raw rate fluctuated slightly
  (90.6% vs. ~93% in earlier runs) from ordinary LLM non-determinism across separate
  generations, not a regression from this fix.

## What this doesn't resolve

- Does not add any cache-invalidation UI or automatic cleanup of orphaned old-keyed
  cache files (`data/cache/scenarios/*.json` from before this fix) — they're gitignored,
  harmless disk usage, and simply never looked up again. No action needed unless disk
  space matters.
- Does not protect against direct tampering of a cache file at its *current* correct
  key — that remains an accepted trust boundary of a local disk cache, not a threat
  this fix is meant to address.
- Does not audit whether other cached artifacts in the project (the RAG embeddings
  index, for instance) have a similar "key doesn't capture everything that determines
  the output" gap — only the scenario-result cache was checked here.
