# Landfall v1.2, Phase 2 result — narrator, verifier, and CLI speak in ranges

## Why this phase exists

v1.2 Phase 1 changed what `engine.py` computes (a point estimate plus an interquartile
range that brackets all three storms' actual damage), but left the narrator, verifier,
and CLI speaking in the single point-estimate number — flagged explicitly as a real,
separate follow-on decision rather than bundled in silently. This phase closes that
gap.

## What changed

- **`landfall/llm/narrator.py`**: `narrate()` now takes `damage_low_usd` and
  `damage_high_usd` instead of a single `total_damage_usd`. The system prompt instructs
  the model to state both bounds as a range ("an estimated $X to $Y in damage") and
  explicitly forbids averaging, splitting the difference, or stating any other single
  damage figure — that would be exactly the kind of invented statistic the groundedness
  verifier exists to catch. The prompt also tells the model not to editorialize about
  the range being wide or unhelpful; the width is real calibration uncertainty (v1.2
  Phase 1), not something to apologize for or smooth over.
- **`landfall/verify/verified_narrator.py`**: `narrate_verified()` takes both bounds;
  `reference_values` is now `[damage_low_usd, damage_high_usd, affected_population] +
  permitted_values` — both bounds are individually permitted, but nothing *between*
  them is, so a model-computed midpoint still gets caught and redacted exactly as
  intended.
- **`landfall/cli.py`**: `cmd_run`'s `_print_result` now prints the point estimate
  alongside the full range; `cmd_narrate` passes `total_damage_usd_range["low"/"high"]`
  instead of the point estimate.
- **`evals/e2_groundedness.py`**: updated to the new three-argument call.

## Verified end to end, not just unit-level

- `landfall run haiyan`: prints `$695,357,863.53 point estimate (range:
  $49,596,773.73 - $2,891,237,057.79)`.
- `landfall narrate odette` (historical replay): *"the estimated total economic damage
  is between $185,091,240.57 to $23,530,739,646.49... 10,541,655 people"* — both figures
  match the cached range exactly, no invented midpoint, groundedness 4/4.
- `landfall narrate rolly` (historical replay): range stated correctly, 4/4 grounded.
- `landfall narrate rolly --offset-km 100 --bearing 0 --intensity-delta 20`
  (counterfactual): correctly labeled as a counterfactual, range stated correctly for
  the perturbed scenario, 4/4 grounded — confirms the counterfactual code path (not just
  historical baselines) carries the range through correctly.

## E2 re-run, because the old numbers no longer reflect what the code does

Re-ran the full E2 groundedness eval (63 briefings, same 21 scenario configs × 3
generations as the original) rather than leave stale numbers under an unqualified
percentage claim:

| | Before (point estimate) | After (range) |
|---|---|---|
| Raw groundedness | 188/220 = 85.5% | 252/270 = 93.3% |
| Final groundedness | 189/189 = 100.0% | 252/252 = 100.0% |

Total claims rose from 220 to 270 (three core reference values — low, high, population
— instead of two, so more numeric claims per briefing on average), and raw groundedness
improved rather than degraded. The same known Phase-4 finding recurred during the
re-run and is not a new problem: a handful of ungrounded claims (`'100.0'`, `'150.0'`,
`'180.0'`) traced to the model restating a counterfactual scenario's own perturbation
inputs (track offset/bearing) embedded in its prompt — true numbers, but not cached
impact-engine output, so the verifier correctly flags and redacts them either way.
Final groundedness is 100.0% by construction regardless (redaction forces it), so this
comparison is really about the raw (pre-verifier) rate, and it moved in the right
direction.

## What this doesn't change

- Per-municipality breakdown is still point-estimate only (median quantile) — ranges
  are a total-damage-only figure, per Phase 1's scope, unchanged here.
- The RAG sitrep interrogator (`rag_answer.py`) is unaffected — it narrates retrieved
  historical facts, not impact-engine damage figures, so it has no point estimate or
  range to update.
